## 3. Architecture decisions

### 3.1 Approach (user-decided)
- In-house. No external CMS. No Sanity/Strapi/Contentful.
- Admin UI lives under `/admin/posts`.
- Content stored in Postgres via Prisma.
- Rich text via **TipTap** (extensions: StarterKit, Link, Image, Placeholder,
  CharacterCount, Persian RTL support via `dir="rtl"` on editor wrapper).

### 3.2 Two content layers (avoid confusion)
**Topics** (static, hand-curated by developer, current system, ~35 pages):
- Stay in TS files (`src/lib/topic-content/*.ts`).
- Authority-style evergreen content (e.g. "Sperrkonto explained").
- Rendered via `topic-route.tsx` at `/fa/<group>/<slug>`.
- **Not migrated to DB** â€” they're code-shaped content with typed
  bullet/highlight/FAQ structures, the editor would be a downgrade.

**Posts** (new, admin-editable, time-ordered):
- News, updates, announcements, opinion pieces, case studies.
- Live in DB.
- Rendered at `/fa/news/<slug>` (and category hub feeds).
- Editor experience: TipTap rich text.

This split keeps the "premium curated authority pages" (topics) and the
"frequent updates" (posts) in their right tools.

### 3.3 URL structure
- `/fa/news` â€” paginated news feed (latest first)
- `/fa/news/<slug>` â€” individual post page
- `/fa/news/category/<slug>` â€” category-filtered feed
- `/fa/news/tag/<slug>` â€” tag-filtered feed (optional, can defer)

Hub URLs (`/fa/exams`, etc.) keep their current topic-driven structure,
**plus** a "Latest news" strip at the bottom showing recent posts in
matching categories.

## 4. Data model

```prisma
model Post {
  id              String       @id @default(cuid())
  slug            String       @unique
  title           String
  seoTitle        String?      // optional override; falls back to title
  metaDescription String?
  excerpt         String?      // short summary for cards
  bodyHtml        String       @db.Text   // TipTap-rendered HTML
  bodyJson        Json?        // TipTap JSON for re-editing
  coverImageUrl   String?      // public/posts/<id>.<ext>
  coverImageAlt   String?
  status          PostStatus   @default(DRAFT)
  publishedAt     DateTime?    // when status flips to PUBLISHED
  authorId        String?
  categoryId      String
  createdAt       DateTime     @default(now())
  updatedAt       DateTime     @updatedAt

  author    User?        @relation(fields: [authorId], references: [id], onDelete: SetNull)
  category  PostCategory @relation(fields: [categoryId], references: [id], onDelete: Restrict)
  tags      PostTag[]    // M2M via PostTagOnPost

  @@index([status, publishedAt])
  @@index([categoryId])
}

enum PostStatus {
  DRAFT
  PUBLISHED
  ARCHIVED
}

model PostCategory {
  id     String @id @default(cuid())
  slug   String @unique           // "exams", "germany-visa", ...
  name   String                   // Persian display name
  order  Int    @default(0)
  posts  Post[]
}

model PostTag {
  id    String       @id @default(cuid())
  slug  String       @unique
  name  String
  posts PostsOnTags[]
}

model PostsOnTags {
  post   Post    @relation(fields: [postId], references: [id], onDelete: Cascade)
  postId String
  tag    PostTag @relation(fields: [tagId], references: [id], onDelete: Cascade)
  tagId  String
  @@id([postId, tagId])
}
```

**Seed:** categories pre-seeded to match brief â€” `exams`, `germany-visa`,
`study-germany`, `work-germany`, `life-germany`, `news-updates`.

## 8. Related-articles algorithm

Simple, no ML:
1. Same category, different post â†’ score +3.
2. Per shared tag â†’ +1.
3. Bonus for recency (e.g., posts within last 30 days â†’ +0.5).
4. Sort by score, take top 4. If fewer than 4 qualify, fill from
   most-recent in the same category.

Runs as a server-side helper at render time. Cache per-post for an hour
(low DB load given posts don't change often).

## 13. Risks

- **TipTap bundle size** (~150KB minified). Admin route only, not loaded
  on public pages. Acceptable.
- **Image upload security** â€” must validate MIME, cap file size, store
  outside DB. Pattern exists in current `/api/documents` flow.
- **HTML sanitization on render** â€” TipTap output is generally clean
  but we should still sanitize on render (DOMPurify or rehype-sanitize)
  to prevent stored XSS if a malicious admin ever exists.
- **Sitemap size** â€” 11 migrations + ~50 topic pages + future posts.
  Sitemap is dynamic, regenerates each request. If we hit 1000+ posts
  someday, split into sitemap index.

## 14. Out of scope (explicit)

- Comments / discussion
- Email subscriptions to news (separate from existing marketing list)

- Author profiles / multi-author byline
- Newsletter generation from latest posts (cool but separate)
